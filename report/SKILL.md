# /report — Browser-Rendered Report Generator

You are a report designer. Take structured data (from other skills, files, or user input), render it as a polished HTML report, then use Puppeteer to screenshot and optionally PDF-export it. Produce client-ready deliverables.

## Arguments

- `/report <file>` — generate HTML report from a markdown/text file
- `/report --from site-audit` — convert the latest site-audit report to HTML
- `/report --from prospect-research` — convert the latest prospect-research brief to HTML
- `/report --from client-brief` — convert the latest client brief to HTML
- `/report --from campaign-check` — convert the latest campaign check to HTML
- `/report --template <name>` — use a specific template (see Templates section)
- `/report --title "Custom Title"` — override the report title

## Phase 1: Identify Source Data

1. Parse the user's input to determine the data source:
   - If `--from <skill>`, find the most recent output file:
     ```bash
     ls -t .pulse/site-audits/audit-*.md 2>/dev/null | head -1
     ls -t .pulse/prospect-research/prospect-*.md 2>/dev/null | head -1
     ls -t .pulse/client-briefs/*.md 2>/dev/null | head -1
     ```
   - If a file path is given, read that file
   - If neither, ask the user what data to report on

2. Read the source file and parse its structure (headings, tables, lists, scores).

3. Create output directory:
   ```bash
   REPORT_DIR=".pulse/reports"
   mkdir -p "$REPORT_DIR"
   ```

## Phase 2: Select Template

Choose based on source or `--template` flag:

### Templates

**executive** (default for client-brief, prospect-research):
- Clean, professional design
- Pulse branding: dark bg `#292929`, coral primary `#ef6552`, warm white `#f8f6f1`
- Embedded base64 logo (heartbeat icon + wordmark)
- Summary card at top with key metrics
- Sections with subtle borders

**technical** (default for site-audit):
- Score gauges and progress bars
- Color-coded severity (red/yellow/green)
- Monospace font for technical data
- Expandable detail sections
- Screenshot grid

**dashboard** (default for campaign-check):
- Card-based metric layout
- Trend indicators (arrows up/down)
- Status badges (OK / WARNING / CRITICAL)
- Table-heavy with alternating row colors

**minimal**:
- Simple, clean typography
- Maximum readability
- Good for printing

## Phase 3: Generate HTML

Build the HTML report with embedded CSS. Reference `report/branding.md` for exact Pulse color values, fonts, and logo URL.

**Logo embedding:** Read the base64 logo from `report/logo-base64.txt` (sibling file in this skill directory). Replace `PULSE_LOGO_BASE64` in the `<img>` src with the full data URI from that file. This keeps reports self-contained with no network dependency.

### Base HTML Structure

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>{{REPORT_TITLE}}</title>
  <style>
    /* Template-specific styles injected here */
  </style>
</head>
<body>
  <div class="report">
    <header class="report-header">
      <div class="branding">
        <img class="logo" src="data:image/png;base64,PULSE_LOGO_BASE64" alt="Pulse" />
        <p class="report-type">{{REPORT_TYPE}}</p>
      </div>
      <div class="report-meta">
        <p class="date">{{DATE}}</p>
        <p class="prepared-for">Prepared for: {{CLIENT}}</p>
      </div>
    </header>
    <main class="report-body">
      <!-- Sections generated from source data -->
    </main>
    <footer class="report-footer">
      <p>Generated by Pulse — pulsemarketing.co</p>
      <p class="confidential">Confidential — Do not distribute</p>
    </footer>
  </div>
</body>
</html>
```

## Phase 4: Content Mapping

Transform source data into HTML sections. Follow these rules:

1. **Headings** (lines starting with `##`) → `<h2>` sections
2. **Tables** (markdown pipe tables) → `<table>` with proper `<th>/<td>`
3. **Bullet lists** → `<ul><li>` elements
4. **Scores/metrics** → Summary card metrics at top
5. **Status labels** (OK/WARNING/CRITICAL) → Badge spans with appropriate class
6. **Percentages** → Score bars where appropriate
7. **Key findings** → Highlight boxes for the top 3

### Content Transformation Rules

- Strip the `====` separator lines from source
- Convert markdown bold (`**text**`) to `<strong>`
- Convert markdown links to `<a>` tags
- Group related metrics into the summary card
- Move "Critical Issues" or "Flagged Issues" near the top, after summary
- If source has screenshots referenced, include `<img>` tags with relative paths

## Phase 5: Render and Capture

1. Write the HTML file to `$REPORT_DIR/report-{source}-{YYYY-MM-DD}.html`

2. Use `puppeteer_navigate` to open the local HTML file:
   - Navigate to `file://<absolute-path-to-html>`

3. Use `puppeteer_screenshot` to capture the rendered report:
   - Name: `report-preview`
   - Width: 900
   - Height: 1200

4. Use `puppeteer_evaluate` to get the full page height for a complete capture:

```javascript
document.documentElement.scrollHeight
```

5. If the page is taller than 1200px, take a second screenshot scrolled down:

```javascript
window.scrollTo(0, document.documentElement.scrollHeight / 2)
```
   Then `puppeteer_screenshot` (name: `report-preview-bottom`, width: 900, height: 1200)

## Phase 6: PDF Export (Optional)

If the user requests PDF (`--pdf`), use `puppeteer_evaluate`:

```javascript
(() => {
  // Trigger browser print to PDF via CSS
  const style = document.createElement('style');
  style.textContent = '@page { size: A4; margin: 20mm; }';
  document.head.appendChild(style);
  return 'Print styles applied — user can Cmd+P or use browser PDF export';
})()
```

Tell the user: "HTML report generated. For PDF: open the HTML file in a browser and use Print → Save as PDF. The print styles are already applied."

## Important Rules

1. All styles must be inline or in `<style>` tags — no external CSS/JS dependencies.
2. The HTML must be self-contained and render correctly when opened from the filesystem.
3. Use relative paths for any referenced images/screenshots.
4. Never include sensitive data (API keys, passwords) in generated reports.
5. If the source file doesn't exist, report the error and ask what to generate from.
6. Keep the HTML under 500 lines — use concise markup. Split into multiple files if needed.
7. Always preview the report via Puppeteer screenshot so the user sees the result.
8. Reports should look professional enough to share with clients directly.
